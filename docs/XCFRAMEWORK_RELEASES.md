# XCFramework Releases

> Purpose: Describe the current XCFramework distribution channel, how to verify it, and how future stable releases will differ.
> Audience: Human developers and automation that consume prebuilt `PgpMobile.xcframework` assets.

## Current Channel

CypherAir publishes a rolling prerelease XCFramework from the current `main` branch as the GitHub release tag `pgpmobile-edge`.

- This channel is updated automatically on every push to `main`.
- It is intended for CI, integration, and manual validation against the current branch tip.
- It is not treated as an immutable stable SDK release.

Each `pgpmobile-edge` prerelease publishes exactly these assets:

- `PgpMobile.xcframework.zip`
- `PgpMobile.xcframework.sha256`
- `pgpmobile-edge.json`

`pgpmobile-edge.json` is machine-readable metadata with these fields:

- `commit_sha`
- `built_at`
- `marketing_version`
- `project_build_number`
- `xcode_version`
- `rustc_version`
- `workflow_url`

This is intentional: a single app marketing version can have multiple Xcode build numbers during development, so XCFramework metadata must carry both values to identify the exact build instance that produced the binary.

## Downloading

Use GitHub CLI:

```bash
gh release download pgpmobile-edge \
    --pattern 'PgpMobile.xcframework.zip' \
    --pattern 'PgpMobile.xcframework.sha256' \
    --pattern 'pgpmobile-edge.json'
```

Extract the XCFramework after verification:

```bash
ditto -x -k PgpMobile.xcframework.zip .
```

## Verification

First validate the checksum:

```bash
shasum -a 256 -c PgpMobile.xcframework.sha256
```

Then verify the GitHub artifact attestation for the zip:

```bash
gh attestation verify PgpMobile.xcframework.zip -R cypherair/cypherair
```

The current rolling prerelease uses GitHub artifact attestations generated by the release workflow. This is distinct from GitHub release attestations that are automatically created for immutable releases.

## Future Stable Releases

Future stable XCFramework releases are reserved for versioned tags of the form:

- `pgpmobile-vX.Y.Z-buildN`

Stable releases should:

- point to a fixed marketing-version + build-number pair instead of only a marketing version
- keep the same asset naming convention where practical
- publish release notes that describe compatibility and consumer expectations
- be candidates for immutable releases once the channel stops overwriting the same tag and assets

## Why Immutable Releases Are Not Enabled Yet

The current `pgpmobile-edge` channel is a rolling prerelease that force-updates the same tag and re-uploads the same asset names on each `main` push.

GitHub immutable releases would block that workflow because, once published:

- the associated tag cannot be moved
- attached assets cannot be modified or deleted

Because of that, immutable releases are intentionally deferred until the repository introduces a separate stable channel such as `pgpmobile-vX.Y.Z-buildN` that does not reuse tags or assets in place.
