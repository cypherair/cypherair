## Implementation Decision

Production Apple Secure Enclave Custody must be implemented as
operation-level custody, not as another way to unwrap complete OpenPGP secret
certificate bytes.

The current shipped architecture remains a software-secret-certificate model:
Rust generates or imports a complete OpenPGP secret certificate, Swift wraps the
secret certificate bytes with Secure Enclave-assisted local protection, and
private workflows later unwrap those bytes before calling Rust. That behavior
is valid for portable software custody, but it is not the target Secure Enclave
custody model described here.

The target model is:

- Rust and Sequoia own OpenPGP packet construction, parsing, policy,
  signature framing, ECDH KDF, AES Key Wrap, session-key handling, streaming
  decrypt, and MDC/AEAD authentication.
- Swift workflow services own product orchestration.
- The Security layer owns Apple Secure Enclave key generation, Keychain handle
  storage, access-control enforcement, role binding, public-key binding, and
  private P-256 operations.
- Secure Enclave custody exposes only private signing and key-agreement
  operations. It never returns a complete secret certificate, private scalar,
  backup handle, KEK, session key, plaintext, or any persistent or loggable
  secret material. If the final ECDH boundary requires Rust to perform the
  OpenPGP KDF and AES Key Wrap unwrap, Swift/Security may provide only the
  minimal in-memory key-agreement operation output needed by Rust.

This document is implementation guidance for later code PRs. It does not
authorize changing sensitive code without the staged roadmap and validation
gates in
[Implementation Roadmap](APPLE_SECURE_ENCLAVE_CUSTODY_IMPLEMENTATION_ROADMAP.md).

## Source Authorities And Red Lines

When documents disagree, use the following precedence for Secure Enclave
custody work:

- [Security Requirements](APPLE_SECURE_ENCLAVE_CUSTODY_SECURITY_REQUIREMENTS.md)
  owns red lines, access-control requirements, validation gates, and release
  blocking security decisions.
- [Architecture Plan](APPLE_SECURE_ENCLAVE_CUSTODY_ARCHITECTURE_PLAN.md) owns
  layer boundaries, resolver/router ownership, and migration requirements.
- [Product Design](APPLE_SECURE_ENCLAVE_CUSTODY_PRODUCT_DESIGN.md) owns user
  commitments, product families, and MVP private-operation scope.
- This document owns the concrete implementation contracts that later PRs
  should follow.
- Current code owns shipped behavior until a reviewed implementation PR changes
  it.

Secure Enclave custody implementation must fail design review if it requires
any of the following:

- importing an existing OpenPGP private key into Secure Enclave custody;
- exporting Secure Enclave private-key material;
- treating a Keychain handle, public key, CryptoKit representation, or locator
  as a recoverable private-key backup;
- storing a software private-key fallback for a Secure Enclave custody key;
- unwrapping, storing, logging, or returning a complete OpenPGP secret
  certificate for a Secure Enclave custody key;
- using one Secure Enclave key for both signing and key agreement;
- accepting a signing handle for ECDH or an ECDH handle for signing;
- weakening payload authentication hard-fail behavior for SEIPDv1/MDC or
  SEIPDv2/AEAD;
- moving OpenPGP packet rules, v4/v6 signature framing, PKESK parsing, message
  format selection, or payload release policy into Swift/Security.

Apple platform documentation also constrains the design. Security-framework
Secure Enclave keys are generated by the Secure Enclave, are limited to 256-bit
elliptic-curve private keys, and cannot import existing private-key material.
Relevant Apple references include
[Protecting keys with the Secure Enclave](https://developer.apple.com/documentation/security/protecting-keys-with-the-secure-enclave),
[kSecAttrTokenIDSecureEnclave](https://developer.apple.com/documentation/security/ksecattrtokenidsecureenclave),
[privateKeyUsage](https://developer.apple.com/documentation/security/secaccesscontrolcreateflags/privatekeyusage),
[biometryAny](https://developer.apple.com/documentation/security/secaccesscontrolcreateflags/biometryany),
[userPresence](https://developer.apple.com/documentation/security/secaccesscontrolcreateflags/userpresence),
[SecKeyCreateSignature](<https://developer.apple.com/documentation/security/seckeycreatesignature(_:_:_:_:)>),
and
[SecKeyCopyKeyExchangeResult](<https://developer.apple.com/documentation/security/seckeycopykeyexchangeresult(_:_:_:_:_:)>).

`userPresence` must not be used as the default Secure Enclave custody policy
because Apple documents it as biometric or device passcode access. The planned
default is `privateKeyUsage + biometryAny`, but the no-passcode-fallback claim
must remain a hardware validation gate because Apple documentation has
platform-specific wording that must be confirmed on real devices.

## Configuration, Custody, And Capability Types

Implementation must keep three concepts separate.

**OpenPGP configuration**

The configuration model describes OpenPGP behavior. It must be able to
represent at least:

- v4 compatible software configuration;
- v6 modern software configuration;
- v4 P-256 compatible Secure Enclave candidate;
- v6 P-256 modern Secure Enclave candidate.

The exact Swift type name is a later implementation choice, but the successor
to `PGPKeyProfile` should behave like a versioned configuration identifier,
not like a custody selector. It must include or derive:

- OpenPGP key version;
- primary signing/certification algorithm;
- encryption subkey algorithm;
- advertised feature flags and message-format preferences;
- compatibility target, such as GnuPG-oriented or RFC 9580-oriented;
- whether software export and S2K are meaningful for that configuration.

`PGPKeyProfile` must not gain a Secure Enclave case. Existing Profile A and
Profile B records may be normalized into configuration plus software custody
during metadata migration.

**Private-key custody descriptor**

The custody descriptor describes how private operations are reached. It should
be versioned from the first migration and should include at least:

- `softwareWrappedSecretCertV1`, representing the current portable software
  secret-certificate custody model;
- `secureEnclaveP256V1`, representing distinct Secure Enclave P-256 signing
  and key-agreement handles.

The custody descriptor must be stored outside display-only key identity fields.
It is not a user-visible profile name and must not define packet format or
message-format policy.

**Operation capability projection**

The capability projection describes what a key can do now. It should be
derived from product policy, configuration, custody, migrated metadata, public
certificate association, handle association, and runtime availability reduced
to user-displayable status.

Capabilities should be expressed at product-operation level, for example:

- public certificate export;
- private-key export;
- revocation artifact export;
- message signing;
- message decryption;
- sign plus encrypt;
- streaming sign/decrypt/encrypt-plus-sign;
- expiry modification and binding refresh;
- key-level revocation;
- selective subkey or User ID revocation;
- contact certification.

Unsupported states must be explicit and non-ambiguous. For example,
private-key export is supported for software custody and unsupported for Secure
Enclave custody.

## Metadata V2

Secure Enclave custody requires a versioned protected key metadata migration.
The destination schema should be treated as metadata v2, even if the final code
uses a different versioning mechanism.

Metadata v2 should normalize each key record into these conceptual fields:

- stable public identity fields already needed by the app, such as fingerprint,
  user ID, expiration, revocation, default-key status, and public certificate
  bytes;
- OpenPGP configuration identifier;
- custody descriptor;
- public certificate digest used to detect stale or mismatched public material;
- signing public-key binding for Secure Enclave custody;
- key-agreement public-key binding for Secure Enclave custody;
- operation capability projection or enough non-secret state to derive it;
- revocation artifact presence and export state;
- private-key export state, including explicit non-exportability for Secure
  Enclave custody;
- recovery state for metadata/handle mismatch, missing handle, corruption, or
  local reset completion.

Metadata v2 must not contain:

- Secure Enclave private-key material;
- complete OpenPGP secret certificates for Secure Enclave custody;
- raw ECDH shared secrets, KEKs, session keys, or decrypted plaintext;
- access-control policy as user-owned data;
- Keychain locators in logs, diagnostics, or user-visible error details.

The migration from current metadata must:

- preserve existing Profile A/B key behavior;
- represent existing keys as software secret-certificate custody;
- never convert an existing software private key into Secure Enclave custody;
- preserve readable source state until the migrated destination is validated;
- fail closed on corrupt committed protected state;
- avoid empty-key-list flashes before authorization;
- update [Persisted State Inventory](PERSISTED_STATE_INVENTORY.md) in the same
  implementation PR that introduces new persistent state.

## Secure Enclave Handle Store

Secure Enclave custody handle storage must be separate from the current
`se-key` / `salt` / `sealed-key` bundle. The current bundle protects complete
software secret certificate bytes. It must not become the Secure Enclave
private-operation-handle model.

The handle store should provide separate lifecycle records for:

- signing handle;
- key-agreement handle;
- role binding for each handle;
- expected public key for each handle;
- expected OpenPGP certificate association;
- cleanup participation for local reset and key deletion;
- recovery classification when metadata and handles disagree.

Each Secure Enclave private key must be generated on-device with the Secure
Enclave token. The creation attributes must include the Secure Enclave token at
creation time and an access control created with:

- `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` unless a later security review
  explicitly chooses a stricter compatible accessibility class;
- `privateKeyUsage`;
- `biometryAny`;
- no default `userPresence`;
- no default `devicePasscode`;
- no user-selectable `biometryCurrentSet` option for v1.

The implementation must use distinct Secure Enclave keys for signing and key
agreement. It must validate both role and public-key binding before any private
operation. A missing handle, wrong role, wrong public key, unavailable
biometry, cancellation, lockout, platform unsupported state, or Keychain error
must fail closed and return a sanitized operation error.

The handle store may keep opaque identifiers needed to find Keychain items, but
those identifiers are sensitive operational state. They must not be logged,
surfaced in UI, or persisted in Rust as OpenPGP secret material.

## Capability Resolver

The capability resolver is a pure policy and projection component. It answers
which configurations and operations are valid before a workflow starts.

Inputs:

- product policy;
- OpenPGP configuration;
- custody descriptor;
- migrated key metadata;
- public certificate and handle association status;
- runtime availability reduced to safe display state;
- feature gate state for hidden/test-only Secure Enclave work.

Outputs:

- valid key-generation choices;
- per-key operation capabilities;
- unavailable states suitable for UI and workflow decisions;
- validation errors for impossible configuration/custody combinations.

The resolver must not:

- prompt for biometrics;
- call Secure Enclave private operations;
- mutate Keychain rows;
- unwrap software secret certificates;
- call Sequoia packet decrypt or signing logic;
- own workflow-specific side effects.

Workflow services should consult the resolver before offering an operation.
The UI should use resolver output to show complete valid product choices rather
than exposing a free-form matrix of profile, algorithm, custody, and
access-control flags.

## Private-Key Operation Router

The private-key operation router is the execution dispatch point. It receives a
resolved product operation and returns a custody-specific route:

- software secret-certificate route for current portable keys;
- Secure Enclave external signer route;
- Secure Enclave ECDH/session-key acquisition route;
- explicit unsupported route.

The router centralizes custody-specific branching so signing, decryption,
encryption, password-message, key-management, and certificate services do not
grow separate custody switches.

The router must enforce:

- no software fallback for Secure Enclave custody;
- no secret-cert unwrap fallback for Secure Enclave custody;
- no unsupported operation downgrade into a software route;
- no bypass around role/public-key binding;
- sanitized error reporting.

The router does not own UI text, metadata migration, Keychain row design,
OpenPGP packet semantics, or payload authentication. Those remain with their
respective layers.

## Rust And UniFFI Boundary

Rust/Sequoia must keep OpenPGP semantics. Swift/Security should expose only
the minimum private P-256 operations needed to finish OpenPGP work.

The Rust boundary should add Secure Enclave-specific APIs rather than forcing
SE custody through existing APIs that accept complete secret certificate bytes.
The exact UniFFI names may change, but the boundary should support these
concepts:

- public certificate construction from P-256 public signing and key-agreement
  keys;
- external signing request created by Rust after building the OpenPGP
  signature context;
- external ECDH request created by Rust after parsing recipient PKESK material;
- callback or operation result that returns only the agreed output required by
  the specific route, not private material;
- stable sanitized error categories for cancellation, unavailable hardware,
  wrong role, wrong public key, unsupported operation, malformed packet,
  authentication failure, and internal failure.

### Signing

The signing path should use a Sequoia `crypto::Signer`-style adapter.

Rust owns:

- signature packet construction;
- v4/v6 signature context;
- v6 salt handling;
- hash algorithm selection;
- digest prefix and digest truncation/padding rules;
- conversion into OpenPGP `mpi::Signature`.

Swift/Security owns:

- loading the expected signing handle;
- validating role and public-key binding;
- checking `SecKeyIsAlgorithmSupported` for the selected ECDSA algorithm;
- performing the Secure Enclave signature operation and returning the agreed
  signing result to Rust.

The external signer must not hash the message again. It signs the digest
provided by Rust for the OpenPGP signature context.

### Decryption

The decrypt path must keep recipient/session-key acquisition separate from
payload decrypt and authentication.

Rust owns:

- PKESK/SKESK iteration and recipient matching;
- OpenPGP ECDH parameters;
- OpenPGP ECDH KDF;
- AES Key Wrap unwrap;
- session-key validation and zeroization;
- SEIPDv1/MDC and SEIPDv2/AEAD payload decrypt;
- read-to-completion and message-processed semantics;
- success-only streaming output commit.

Swift/Security owns:

- loading the expected ECDH handle;
- validating role and public-key binding;
- performing the Secure Enclave P-256 key agreement operation;
- returning only the minimal in-memory key-agreement operation output needed by
  Rust.

Apple X9.63 KDF algorithms are not the OpenPGP ECDH KDF. The production design
should prefer a boundary where Rust performs OpenPGP KDF and unwrap. If an
intermediate implementation uses a fixed-session-key helper, it must still
preserve recipient binding, authenticated payload release, and tamper hard-fail
behavior.

The Phase 4 POC response-file bridge that wrote raw shared secrets to JSON is
not production acceptable. Production must not write shared secrets, KEKs,
session keys, or plaintext to temporary files or diagnostics.
Any key-agreement operation output that crosses into Rust must be zeroized,
must never be persisted, must never be logged, must never be exposed to UI, and
must never be routed through the old response-file bridge.

## Operation Semantics

Every workflow that currently uses complete secret certificate bytes must have
an explicit Secure Enclave custody behavior before product launch.

| Operation | Software custody behavior | Secure Enclave custody behavior |
| --- | --- | --- |
| Key generation | Generate/exportable software secret cert, then wrap locally. | Generate distinct SE signing/ECDH keys, build P-256 OpenPGP public certificate, store metadata/handles, and never create a secret cert fallback. |
| Private-key import | Supported as software custody. | Unsupported. Existing private keys cannot be imported into Secure Enclave custody. |
| Public-key export/share | Export public material. | Export public material. Custody is not private-key backup. |
| Private-key export/backup | Supported where software custody policy allows it. | Unsupported. UI must say non-exportable/device-bound. |
| Message signing | Unwrap secret cert and use Rust/Sequoia keypair. | Route through external signer; no secret cert unwrap. |
| Encrypt plus sign | Encrypt to recipients and sign through software route. | Encrypt to recipients and sign through external signer. |
| Password-message optional signing | Optional software signing route. | Optional external signer route. |
| Message decrypt | Unwrap secret cert and use Rust/Sequoia decrypt helper. | Route PKESK/session-key acquisition through SE ECDH; payload decrypt/auth remains Rust/Sequoia. |
| Streaming file operations | Current success-only temp/output contract. | Same success-only contract; private signing/ECDH routes are external operations. |
| Expiry modification/binding refresh | Mutate secret cert through Rust. | Rebuild/bind public certificate state by external signing; unsupported until this can be done without private-key export. |
| Key-level revocation | Use secret cert to generate/export revocation artifact. | Generate/export revocation artifact through external signing; artifact is not private-key backup. |
| Selective subkey/User ID revocation | Use secret cert after public selector validation. | Use external signing after public selector validation; unsupported until no-fallback path exists. |
| Contact certification | Use secret cert signer. | Use external signer; unsupported until covered by tests and UI. |
| Mode switching Standard/High Security | Rewrap software custody bundle. | Not applicable. SE custody v1 does not rewrap access policy in place. |

Unsupported Secure Enclave operations must fail closed and show product-level
unavailable state. They must not silently fall back to software custody.

## Error Handling And Logging

Secure Enclave custody errors should be classified for product behavior without
revealing sensitive state.

Recommended categories:

- unsupported platform;
- Secure Enclave unavailable;
- required biometric authentication unavailable;
- authentication canceled;
- authentication failed or locked out;
- handle missing;
- handle role mismatch;
- handle public-key mismatch;
- metadata/public certificate mismatch;
- OpenPGP packet malformed;
- private operation failed;
- payload authentication failed;
- operation unsupported for custody.

Errors, logs, diagnostics, test output, and UI must not include:

- plaintext;
- private-key material;
- complete secret certificate bytes;
- ECDH shared secrets;
- KEKs;
- session keys;
- Keychain locators;
- stable fingerprints where not already part of an intentional user-visible
  public-key surface;
- temporary file paths that reveal sensitive operation state.

Swift transient `Data` that contains sensitive material must be explicitly
zeroed with `Data.resetBytes(in:)`. Rust sensitive buffers must use the
existing zeroize discipline.

## Test Contracts

Implementation PRs should add mockable tests before hardware-dependent tests.

Mockable Swift tests:

- metadata v2 migration from existing Profile A/B to configuration plus
  software custody;
- legal and illegal configuration/custody combinations;
- resolver output for supported, unsupported, missing, and unavailable states;
- router dispatch for software custody, SE signer, SE ECDH, and unsupported
  routes;
- no software fallback and no secret-cert unwrap fallback for SE custody;
- wrong role, wrong public key, missing handle, and metadata/handle mismatch;
- export UI/state rejects private-key export for SE custody;
- local reset and deletion cleanup classification.

Mockable Rust tests:

- fake external P-256 signer receives the correct digest and does not double
  hash;
- fake external ECDH covers OpenPGP KDF, AES-KW unwrap, v3/v6 PKESK behavior,
  wrong recipient, tampered ephemeral public point, tampered wrapped session
  key, and malformed packets;
- tampered SEIPDv1/MDC and SEIPDv2/AEAD payloads return no plaintext;
- streaming failures do not leave the final output file.

Hardware evidence:

- real Secure Enclave P-256 signing and verification on iOS, iPadOS, macOS,
  and visionOS where supported;
- real Secure Enclave ECDH decrypt for v4 SEIPDv1/MDC and v6 SEIPDv2/AEAD;
- authentication cancellation, lockout, unavailable biometry, enrollment
  changes, missing handles, local reset, and public-binding mismatch;
- validation that `privateKeyUsage + biometryAny` does not permit device
  passcode fallback on the supported platform set before any release claim.

Interop evidence:

- v4 P-256 public certificate imports into GnuPG;
- GnuPG verifies SE-generated v4 signatures;
- GnuPG encrypts to the v4 SE public certificate using the expected PKESK v3
  ECDH plus SEIPDv1/MDC shape;
- CypherAir decrypts and verifies GnuPG-originated messages through the
  production SE boundary;
- v6 P-256 validates RFC 9580 / AEAD behavior without GnuPG compatibility
  claims unless later evidence supports them.
