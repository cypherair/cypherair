## Disposition plan

**Fix now (7)** — sound, self-contained, low residual-risk; remediation PR to follow:

- `release-secrets-env` (high) — scrub `GITHUB_PAT` / `ASC_*` from untrusted build subprocesses (rustup/brew/cargo/pip).
- `sqlcipher-zip-extract` (high) — extract to a temp dir, allowlist entries (reject `..`/absolute/symlink/extra), then atomically move.
- `session-digest` (low) — `umask 077` so `/tmp` digests aren't world-readable.
- `perl-inject` (info) — pass the path via `%ENV` into a single-quoted Perl program (both fixture-generator copies).
- `envelope-forgery` (medium) — after unwrap, bind the *actual* certificate identity to the requested route before any consumer uses it.
- `pq-badge` (medium) — require the SEIP container be observed before returning `FullyPostQuantum`/`Mixed`; fail closed on a truncated prefix.
- `resume-race` (medium) — hold the cosmetic privacy cover until the foreground re-auth decision resolves.

**Flag for maintainer (2)** — real, but the load-bearing fix is architectural and needs a design/wording decision, so not auto-remediated:

- `release-attest-oracle` (high) — the honest full fix is builder-emitted provenance, which Xcode Cloud cannot produce (no GitHub OIDC attestation from the builder). Available in-workflow hardening (asset-vs-manifest digest cross-check) is weak because the manifest is itself an unsigned downloaded asset; the durable options are a source-bundle-from-signed-commit rebuild/compare and reframing `docs/RELEASE.md` to label the stable attestation as a *publication witness* (which the workflow header already says internally). Presupposes a compromised release/Xcode-Cloud publish token.
- `merge-artifacts` (high, partial) — official release paths (WF1/WF2 + `stable-release-attest.yml`) rebuild + shasum + attest, so shipped releases are **not** exposed; the residual gap is the break-glass local App Store archive, where `validate_app_store_candidate_release.py` never hashes the local `PgpMobile.xcframework` against the attested stable binary. The load-bearing fix (per-slice hashes in the arm64e manifest + a break-glass hash-pin) changes the manifest schema, and the 'rebuild-not-copy' hygiene half conflicts with the maintainer's established worktree→main copy fast-path — both are maintainer calls.

**No fix (11):**

- *Already fixed (1):* `hidden-recipients` — fixed ~1h after detection in `513fc75` (ancestor of HEAD); the chooser now keeps the selection count filter-independent and shows an inline "N hidden by filter / Show All" notice.

- *False positive (2):* `untrusted-cert` — an OpenPGP certification is a factual "a signature exists" statement, shown under an attributed "OpenPGP Certification" badge; it is fully separate from the manual-fingerprint trust gate and never feeds recipient selection. `gpg-sort-v` — macOS 27's BSD `sort` supports `-V` (verified directly); the required interop job has been green across many merges.

- *Wont-fix by policy (8):* `reset-larights`, `authmode-downgrade`, `contacts-rollback`, `reset-metadata-rows`, `grace-period`, `aad-length`, `mocks-in-target`, `dmk-migration` — each is either a deliberate documented design decision or a legacy-only impact (materializes only on a maintainer dev device upgraded across a cutover); the remedy each finding recommends (re-add migration/compat/reset-cleanup for a retired format) is exactly what the total-removal doctrine forbids. Per-finding reasoning below.

## Implementation note (fix placement)

Fix placement is **not** constrained to one language side. Where a finding spans the Rust↔Swift bridge (notably `envelope-forgery` and `pq-badge`), the remediation PR should choose the placement that is most architecturally sound and maintainable for the project — Swift chokepoint, Rust engine surface, or a hybrid — rather than defaulting to whichever side is smaller. For `envelope-forgery`, one clean single-point option is the Swift unwrap chokepoint (`PrivateKeyAccessService.unwrapPrivateKey`, through which all 13 software-secret-certificate consumers funnel) using the existing `parseKeyInfo` FFI; a Rust-side identity binding on the signing/certification surfaces (mirroring the `select_external_signing_key` chokepoint added in #590) is the alternative. The decision is deferred to implementation.

### `sqlcipher-zip-extract` — SQLCipher zip is extracted into repo before path validation

- **Severity (Codex):** high · **Detected:** 2026-06-27 · **Scanned commit:** `80111d3`
- **Verdict:** **still-present** (high confidence) · **Security-critical:** yes · **Fix size:** small · **Disposition:** **Fix now**
- **Paths (as recorded):** scripts/restore_sqlcipher_xcframework.sh | scripts/validate_sqlcipher_xcframework.py | .github/workflows/xcframework-edge-release.yml
- **Evidence (current HEAD):**
  - `scripts/restore_sqlcipher_xcframework.sh:228` — ditto -x -k "$WORK_DIR/SQLCipher.xcframework.zip" "$REPO_ROOT" extracts the pinned zip directly into the repository root with no per-entry inspection; ditto does not itself reject `..`/absolute paths or symlinks in zip entries.
  - `scripts/restore_sqlcipher_xcframework.sh:235` — validate_sqlcipher_xcframework.py runs only AFTER extraction (line 235) — confirming the finding's before/after ordering claim.
  - `scripts/validate_sqlcipher_xcframework.py:310` — validate_xcframework only asserts that expected SQLCipher slice files/headers EXIST with expected sha256; it never enumerates zip entries, never rejects extra repo-relative files, `..`, absolute paths, or symlinks. No function in the script inspects the zip's file list at all.
  - `.github/workflows/xcframework-edge-release.yml:189` — PgpMobile.xcframework is built at step 'Build XCFramework...' (line 189-192) BEFORE the SQLCipher restore step (line 193-197), then repackaged at line 235 (ditto -c -k ... PgpMobile.xcframework). A SQLCipher zip carrying `PgpMobile.xcframework/...` entries would overwrite the freshly built framework before it is packaged — the exact path the finding describes.
  - `.github/workflows/xcframework-edge-release.yml:345` — MITIGATING: the upload-artifact `path:` is an explicit allowlist (XCFRAMEWORK_ZIP, CHECKSUM, ARM64E_MANIFEST, RELEASE_METADATA, release-notes.md), so stray files dropped at repo root are NOT published unless they overwrite one of those exact names or the contents of PgpMobile.xcframework that feeds line 235.
  - `scripts/restore_sqlcipher_xcframework.sh:212` — MITIGATING: the zip is sha256-verified (verify_asset_hash against EXPECTED_ZIP_SHA from the checked-in pin) BEFORE extraction, and under --require-attestation (used by the workflow, line 197) also gh-attestation-verified against the SQLCipher signer workflow. So the zip cannot be substituted at restore time — its bytes are fully pinned/attested.
  - `third_party/sqlcipher-xcframework.pin.json:1` — Pin file is git-tracked and carries the zip sha256; altering which zip is restored requires editing this in-repo pin (repo write), at which point an attacker could edit the workflow directly. So the residual attack requires the pinned SQLCipher build repo itself to emit a zip with hidden extra entries.

**Verdict reasoning:** Verdict: the finding is technically accurate and the described mechanism is unchanged at HEAD (22b2759). The restore script does `ditto -x -k <zip> "$REPO_ROOT"` (line 228) BEFORE running validation (line 235), and the validator only checks that the EXPECTED SQLCipher files exist with expected hashes — it never enumerates zip entries or rejects `..`/absolute-path/symlink/extra entries, so a zip with stray paths (e.g. `PgpMobile.xcframework/...`) would be written into the repo, and the edge-release workflow builds PgpMobile.xcframework (line 189) before the restore (line 193) and repackages it (line 235), matching the exact poisoning path described. So this is a real defense-in-depth (zip-slip / extraneous-write) gap and I classify it still-present. HOWEVER, the finding's 'high' severity overstates real-world exploitability: the SQLCipher zip is not attacker-controlled at restore time — its sha256 is pinned in the git-tracked third_party/sqlcipher-xcframework.pin.json and verified before extraction (line 212), and the workflow runs with --require-attestation, which additionally gates on an immutable release plus gh attestation from the pinned SQLCipher signer workflow. To land a malicious zip an attacker must either (a) get the in-repo pin changed (repo write / merged PR — at which point they could edit the workflow itself), or (b) compromise the same-org SQLCipher build repo (cypherair/sqlcipher-xcframework) to emit an attested zip that hides extra entries while its SQLCipher slices still match. The upload-artifact step also uses an explicit path allowlist, so only overwrites of the named assets or of PgpMobile.xcframework's contents propagate. Net: genuine CI supply-chain hardening worth doing (extract to temp, allowlist entries, then atomically move), but practical risk is medium/low rather than high, and it is fully gated behind trusted-publisher + repo-write boundaries. Not legacy-data related, so total-removal doctrine does not apply. No overlap with the 2026-07-06 campaign (#609/#611/#612) or WCR items, which are app-side crypto/DoS, not CI/release scripts. security_critical=true (release/CI supply-chain integrity).

**Proposed fix:** Harden scripts/restore_sqlcipher_xcframework.sh to extract-then-inspect-then-move instead of extracting straight into REPO_ROOT. Concretely, replace lines 226-228: (1) create a fresh temp dir under WORK_DIR; (2) before/instead of ditto-to-root, enumerate the zip's entries (e.g. `python3 -c` with zipfile, or `unzip -Z1`) and fail closed if ANY entry is an absolute path, contains a `..` component, is a symlink, or does not live under the single top-level `SQLCipher.xcframework/` prefix; (3) `ditto -x -k` into the temp dir; (4) reject any extracted symlinks; (5) atomically `mv`/`ditto` only the validated `SQLCipher.xcframework` into REPO_ROOT (after `rm -rf` of the old one). The sidecar files (manifest, privacy, release.json, sha256) already come from WORK_DIR via individual `cp` at lines 230-233, so the zip only needs to yield SQLCipher.xcframework/ and the allowlist can be strict. Optionally add the same entry-enumeration guard as a preflight in validate_sqlcipher_xcframework.py so the checked-in validator (not just the shell script) enforces it. No behavior change for a well-formed release.

### `release-attest-oracle` — Release attestation signs untrusted published assets

- **Severity (Codex):** high · **Detected:** 2026-06-18 · **Scanned commit:** `ee115c1`
- **Verdict:** **still-present** (high confidence) · **Security-critical:** yes · **Fix size:** medium · **Disposition:** **Flag — architectural**
- **Paths (as recorded):** .github/workflows/stable-release-attest.yml | docs/XCFRAMEWORK_RELEASES.md (removed→docs/RELEASE.md)
- **Overlap:** Not in WCR-01..22 set; distinct from #589/#590 signing chokepoint. No open GitHub issue found tracking it.
- **Evidence (current HEAD):**
  - `.github/workflows/stable-release-attest.yml:72` — `sha256sum -c "$XCFRAMEWORK_CHECKSUM"` -- the checksum file PgpMobile.xcframework.sha256 is itself a downloaded release asset (env line 33; downloaded lines 54/61). The check is circular: an attacker who rewrites the release publishes both the malicious zip and a matching .sha256, so this proves nothing about origin.
  - `.github/workflows/stable-release-attest.yml:116` — The only added binding since the flagged commit checks manifest.commitSHA == signed tag's target commit. But CypherAir-compliance-manifest.json is an unsigned downloaded asset and commitSHA/releaseTag are public values, so an attacker forges them trivially. This does NOT bind the XCFramework bytes to the signature.
  - `.github/workflows/stable-release-attest.yml:149` — binaryAssets from the manifest are inspected ONLY to reject mirrored SQLCipher files (fileName startswith 'SQLCipher'). The manifest's binaryAssets[].sha256 for the XCFramework is never compared against the downloaded zip -- no cross-asset digest check at all.
  - `.github/workflows/stable-release-attest.yml:181` — actions/attest generates GitHub build-provenance over the six downloaded (untrusted) assets after only tag-signature + circular-checksum + forgeable-manifest checks -- i.e. it attests attacker-controllable bytes, acting as a signing oracle for any already-SSH-signed stable tag.
  - `.github/workflows/stable-release-attest.yml:8` — Header comment self-discloses the limitation: 'publication witness ... not in-process build provenance' and 'tamper alarm, not a release gate' -- maintainers are aware the semantics are weaker than real provenance.
  - `docs/RELEASE.md:82` — Consumer verification still directs `gh attestation verify PgpMobile.xcframework.zip --signer-workflow .../stable-release-attest.yml` with no caveat that this is only a publication witness -- the over-trust the finding warns about.
  - `.github/workflows/xcframework-edge-release.yml:192` — Contrast: the edge workflow BUILDS from source in-run (`./build-xcframework.sh --release`) and attests those bytes, giving genuine build provenance. Only the stable path is a post-hoc witness over externally-published bytes.
  - `scripts/build_compliance_release_assets.py:153` — The compliance manifest records fileName+sha256 for each binary asset, so a byte-digest binding exists in principle -- but nothing binds that manifest to the signed git tag beyond the forgeable commitSHA field.

**Verdict reasoning:** Plain-language verdict: The finding holds at current HEAD (main 22b2759). stable-release-attest.yml still generates a GitHub build-provenance attestation over assets it merely downloads from the mutable published release; it never rebuilds the XCFramework nor compares the bytes to any digest cryptographically bound to the SSH-signed tag. The three gates it does run are all defeatable by an actor who can publish/modify the release: the checksum step is circular (the .sha256 is a co-published asset), and the manifest-commit check added after the flagged commit (commit 9137226) is cosmetic because the compliance manifest is an unsigned downloaded asset whose commitSHA/releaseTag are public and trivially forged. So an attacker with a compromised release/Xcode-Cloud publish token can, for any existing SSH-signed stable tag that lacks a published release (or by deleting/recreating one), upload a malicious PgpMobile.xcframework.zip plus a matching .sha256 and forged manifest and obtain a valid official attestation -- a signing oracle, exactly as described. This is NOT a legacy/data-format issue, so the total-removal policy does not apply; it is a live CI supply-chain integrity weakness (security_critical). Two mitigating nuances for severity: it presupposes an already-serious compromise (a valid release-publishing token), and the workflow header comment already self-labels the job as a 'publication witness, not in-process build provenance' -- but that caveat is internal and docs/RELEASE.md still tells consumers to `gh attestation verify` against this workflow without qualification. Because the arm64e binary isn't reproducible from source, the honest full fix is architectural (emit provenance from the builder or downgrade consumer trust language); the minimal practical hardening is the source-bundle-from-signed-commit binding plus manifest-digest cross-check and a docs reframe.

**Proposed fix:** Within stable-release-attest.yml, harden the byte binding: (1) replace the self-referential `sha256sum -c` with a comparison of the downloaded XCFramework zip (and the other binary assets) sha256 against the manifest's binaryAssets[].sha256, enforcing cross-asset internal consistency; (2) bind at least the source bundle to the SSH-signed commit by checking out the tag's target commit inside the workflow and regenerating CypherAir-source-bundle.tar.zst via scripts/build_compliance_release_assets.py, then comparing its digest to the published source bundle; (3) update docs/RELEASE.md (lines ~81-83) to label the stable XCFramework attestation explicitly as a publication witness whose trust anchor is the SSH-signed tag + WF1's in-run shasum chain, not independent build provenance. Note: because the arm64e XCFramework binary is not deterministically reproducible from source, fully eliminating the signing-oracle (attesting only builder-produced bytes) requires emitting provenance from the builder itself -- a larger design change, since Xcode Cloud cannot emit GitHub OIDC attestations.

- **Severity (Codex):** high · **Detected:** 2026-06-18 · **Scanned commit:** `4404048`
- **Verdict:** **still-present** (high confidence) · **Security-critical:** yes · **Fix size:** small · **Disposition:** **Fix now**

- **Evidence (current HEAD):**
  - `scripts/build_apple_arm64e_xcframework.sh:6` — Top-level scrub is `unset GH_TOKEN GITHUB_TOKEN` only — the secret actually in use (GITHUB_PAT) and the App Store Connect keys (ASC_ISSUER_ID/ASC_KEY_ID/ASC_PRIVATE_KEY/ASC_PRIVATE_KEY_PATH) are NOT unset, so every cargo/rustc/build.rs subprocess spawned by the build inherits them.

  - `ci_scripts/ci_post_clone.sh:75` — WF1 runs untrusted installers with the workflow-scoped secret env present: rustup `curl --proto ... | sh` (line 75), `brew install` (line 60), `cargo +stable install cargo-audit` (line 97), `cargo audit` (line 98), then `./build-xcframework.sh --release` (line 107). GITHUB_PAT is required by this workflow (line 65) so it is configured as a workflow env var and is in-scope for all of these before it is ever consumed.
  - `ci_scripts/ci_post_clone.sh:17` — Header comment claims `GitHub tokens are intentionally kept out of the Rust/xcframework build subprocesses (the build script unsets them)` — the stated mitigation is incomplete: it unsets GH_TOKEN/GITHUB_TOKEN, not the GITHUB_PAT/ASC_* actually present, so the comment overstates the protection.

  - `scripts/asc_start_build.py:84` — Confirms the ASC secret names read from the environment: ASC_PRIVATE_KEY (line 84), ASC_PRIVATE_KEY_PATH (line 87), ASC_ISSUER_ID (line 102), ASC_KEY_ID (line 103) — these are the values left exposed by the incomplete scrub.
  - `scripts/build_apple_arm64e_xcframework.sh:6` — git log -S confirms the `unset GH_TOKEN GITHUB_TOKEN` line was introduced by 0db7906 (2026-05-31, `fix: harden release workflow supply chain`), which predates the finding commit 4404048 (2026-06-17); no commit in 4404048..HEAD added GITHUB_PAT/ASC_* scrubbing, so the gap is unchanged at HEAD.

**Verdict reasoning:** Verdict: STILL-PRESENT and legitimate. CypherAir's Xcode Cloud release path (ci_scripts/ci_post_clone.sh + ci_post_xcodebuild.sh) requires GITHUB_PAT (repo contents:write) and App Store Connect API keys (ASC_ISSUER_ID/ASC_KEY_ID/ASC_PRIVATE_KEY[_PATH]) as workflow-scoped environment variables. In Xcode Cloud those secret env vars are inherited by every ci_script and every child process, with no per-step scoping. The build hardening that exists (build_apple_arm64e_xcframework.sh unsetting GH_TOKEN/GITHUB_TOKEN) scrubs the wrong names — the GH CLI defaults — not the GITHUB_PAT/ASC_* the project actually injects, so a malicious rustup installer (`curl | sh`), Homebrew formula, cargo-audit or dependency build.rs, or pip package running during the build can read the PAT/ASC key from its environment and exfiltrate it. That is exactly the supply-chain boundary the finding describes and it is unchanged from the finding commit to HEAD (the only relevant commit, 0db7906, predates the finding). This is CI/release supply-chain integrity, not app-runtime code, and it is NOT legacy-scoped — the total-removal doctrine does not apply, so it is a real fix rather than wont-fix-by-policy. No existing issue/PR covers it (the 2026-07-06 campaign issues #609/#611/#612 and the WCR items are unrelated; the WHOLE_CODEBASE_REVIEW doc is no longer in the tree). Residual-risk note: the release still has strong compensating controls (SSH-signed annotated-tag revalidation, release-immutability guard, sha256 verification, arm64e manifest gate), which limit but do not eliminate the impact of a stolen contents:write PAT (it can still create/delete/alter releases and draft assets) or a stolen ASC key (it can trigger Xcode Cloud build runs and read repo git refs). The fix is a mechanical extension of the scrub pattern the maintainer already established, so it should be low-friction to land.

**Proposed fix:** Extend the existing scrub pattern to cover the secrets actually present. (1) In scripts/build_apple_arm64e_xcframework.sh line 6, change to `unset GH_TOKEN GITHUB_TOKEN GITHUB_PAT ASC_ISSUER_ID ASC_KEY_ID ASC_PRIVATE_KEY ASC_PRIVATE_KEY_PATH` (this alone removes them from all cargo/rustc/build.rs subprocesses of the xcframework build; the eight `env -u` wrappers can also add `-u GITHUB_PAT -u ASC_ISSUER_ID -u ASC_KEY_ID -u ASC_PRIVATE_KEY -u ASC_PRIVATE_KEY_PATH` as belt-and-suspenders). (2) In ci_scripts/ci_post_clone.sh, the untrusted install/build phase (ensure_homebrew_formula, ensure_rust_stable rustup pipe, cargo install cargo-audit, cargo audit, ./build-xcframework.sh) runs before GITHUB_PAT is consumed at require_gh_auth (line 112) and never needs ASC_*, so run that phase with the secrets removed — e.g. wrap those calls in `env -u GITHUB_PAT -u ASC_ISSUER_ID -u ASC_KEY_ID -u ASC_PRIVATE_KEY -u ASC_PRIVATE_KEY_PATH ...`, or execute the install/build block in a subshell that unsets them, restoring GITHUB_PAT only for require_gh_auth. (3) In ci_post_xcodebuild.sh, run the `pip install pyjwt[crypto]` (line 219) with the same `env -u` scrub since it precedes asc_start_build.py. Update the header comment on ci_post_clone.sh lines 17-18 to name the secrets now scrubbed.
